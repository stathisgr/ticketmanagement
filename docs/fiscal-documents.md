# Οδηγός: Μοντέλο Παραστατικών & Ειδών

Γενικό μοντέλο για να διαχειρίζεται μια εφαρμογή **τι** εκδίδει (παραστατικά) και **πώς**
αριθμεί, ανεξάρτητα από τον πάροχο. Αναφορά: `server/src/fiscal/docs.ts`, `issue.ts`.

> Δες μαζί: [mydata-provider.md](./mydata-provider.md) (σύνδεση παρόχου & σφάλματα).

---

## 1. Είδη (kind): Υπηρεσία vs Προϊόν

Κάθε «είδος» (ticket type / προϊόν) έχει `kind`:

- `0` = **Υπηρεσία** (π.χ. εισιτήριο) → ΑΠΥ (απόδειξη παροχής υπηρεσιών).
- `1` = **Εμπορικό προϊόν** (λιανική) → ΑΛΠ (απόδειξη λιανικής).

Η μηχανή έκδοσης διαλέγει παραστατικό βάσει του είδους της πώλησης (αν περιέχει έστω ένα
προϊόν → λιανική προϊόντων· αλλιώς υπηρεσία).

**Παγίδα:** σε κάποια context επιτρέπονται μόνο υπηρεσίες (π.χ. «θέαμα» δέχεται μόνο
εισιτήρια, όχι προϊόντα). Φίλτραρε στη λίστα επιλογής **και** στο save (να μη γλιστρήσει
προϊόν σε λάθος context).

## 2. Ενιαία λίστα παραστατικών (`docs.list`)

Αντί για χωριστά πεδία ανά τύπο, κράτα **μία λίστα** όπου κάθε γραμμή:

```
{ id, label, enabled, role:'sale'|'credit', for:'service'|'product',
  counterpart:'retail'|'invoice', creditDocId,
  invoiceTypeId, series, aaStart, incomeCatId, incomeValId, vatExemptionId,
  paymentCashId, paymentCardId, acquirerId, paymentStatus }
```

Τυπικά:

| Παραστατικό | myDATA τύπος | Σειρά | for | role |
|---|---|---|---|---|
| ΑΠΥ (υπηρεσίες) | 20 (11.2) | ΑΠΥ | service | sale |
| ΠΑΠΥ (πιστωτικό υπηρ.) | 22 (11.4) | ΠΑΠΥ | service | credit |
| ΑΛΠ (λιανική προϊόντων) | 19 (11.1) | ΑΛΠ | product | sale |
| ΠΛΠ (πιστωτικό λιαν.) | 22 (11.4) | ΠΛΠ | product | credit |

Κανόνες επιλογής: κάθε `sale` δένει με ένα `credit` ίδιου `for` (μέσω `creditDocId`).
Πρέπει να υπάρχει τουλάχιστον ένα ενεργό sale υπηρεσιών.

## 3. Αρίθμηση ΑΑ (ανά σειρά)

- Επόμενο: `aaNum = max(aaStart, MAX(αα διαβιβασμένων στη σειρά) + 1)`.
- Το `aaStart` («Τρέχον/Αρχικός ΑΑ») είναι **πάτωμα** — αν το ανεβάσεις, η αρίθμηση συνεχίζει από εκεί.
- **Persist:** μετά από κάθε επιτυχή έκδοση, γράψε το ΑΑ πίσω στο config του παραστατικού (`aaStart`), ώστε να είναι **ορατό & ρυθμιζόμενο** από τις Ρυθμίσεις. Το `MAX` της βάσης παραμένει το «δίχτυ» που εμποδίζει σύγκρουση ακόμη κι αν το config μείνει πίσω.
- **Αποτυχίες δεν καίνε αριθμούς:** μόνο τα `status='transmitted'` μετράνε στο MAX.

```sql
SELECT MAX(CAST(aa AS INTEGER)) FROM fiscal_documents
 WHERE role='sale' AND series=? AND status='transmitted';
```

## 4. Κανονικοποίηση σειράς (ομόγραφα) — ΚΡΙΣΙΜΟ

Οπτικά ίδιοι αλλά διαφορετικοί χαρακτήρες (λατινικό `Y` vs ελληνικό `Υ`) δημιουργούν
**δύο διαφορετικές σειρές** → μηδενισμός/διπλασιασμός αύξοντα. Λύση:

1. **Forward:** κανονικοποίησε κάθε σειρά σε μία γραφή (π.χ. ελληνικά) πριν την έκδοση.
2. **Backfill:** one-time migration που κάνει relabel τις παλιές rows (`fiscal_documents.series`, και όπου αλλού αποθηκεύεται η σειρά) Latin→Greek.

```ts
const LAT2GR = { A:'Α',B:'Β',E:'Ε',Z:'Ζ',H:'Η',I:'Ι',K:'Κ',M:'Μ',N:'Ν',O:'Ο',P:'Ρ',T:'Τ',Y:'Υ',X:'Χ' };
const toGreek = (s) => [...String(s||'')].map(c => LAT2GR[c.toUpperCase()] || c).join('');
```

## 5. Auto-retry σε διπλό ΑΑ

Τύλιξε την κλήση έκδοσης σε βρόχο: αν το σφάλμα είναι «διπλό ΑΑ / ήδη αποθηκευτεί»
(1114/1196), `aaNum += 1` και ξαναδοκίμασε. Εφάρμοσέ το **και στο πιστωτικό**.

## 6. Πιστωτικά

- Λιανική: νέο παραστατικό με **θετικά** ποσά, χωρίς CorrelatedMarks.
- Τοπικός αντιλογισμός: ακύρωση γραμμών + αρνητική κίνηση ταμείου + μείωση ΦΠΑ.

## Παγίδες

- Μην βασίζεσαι σε μετρητή στο config χωρίς το `MAX` της βάσης ως δίχτυ.
- Μην αφήνεις προϊόντα να μπαίνουν σε contexts «μόνο υπηρεσιών» (φίλτρο σε UI **και** save).
- Πρόσεξε τα ομόγραφα σε **κάθε** σημείο που αποθηκεύεται η σειρά (όχι μόνο στο κύριο table).
