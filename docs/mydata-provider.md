# Οδηγός: Πάροχος ηλεκτρονικής τιμολόγησης / myDATA (RapidSign / RBS)

Γενικός οδηγός σύνδεσης μιας εφαρμογής με **πάροχο ηλεκτρονικής τιμολόγησης** για
αυτόματη έκδοση παραστατικών (ΑΠΥ, ΑΛΠ, Τιμολόγια, Πιστωτικά) και διαβίβαση στη **myDATA (ΑΑΔΕ)**.
Αναφορά υλοποίησης: `server/src/fiscal/` (`rapidsign.ts`, `issue.ts`, `docs.ts`).

> Δες μαζί: [fiscal-documents.md](./fiscal-documents.md) (μοντέλο παραστατικών & αρίθμηση).

---

## 1. Τι κάνει η σύνδεση

Με ενεργή λειτουργία «έκδοση μέσω παρόχου», κάθε **πώληση** εκδίδει παραστατικό →
παίρνει **ΜΑΡΚ / UID / QR (myDATA)** → αποτυπώνονται στην απόδειξη/εισιτήριο. Κάθε
**ακύρωση/επιστροφή** εκδίδει **Πιστωτικό**.

- **Hosts:** dev `https://dev.rapidsign.com.gr` · prod `https://app.rapidsign.com.gr`
- **Auth flow:** `Authorize` (username/password/activationCode) → parentToken →
  `RefreshToken` (header `ParentToken`) → bearer token για όλες τις κλήσεις.

## 2. Διαπιστευτήρια (από τον πάροχο)

| Πεδίο | Περιγραφή |
|---|---|
| username / password | API credentials |
| activationCode | Κωδικός ενεργοποίησης εγκατάστασης |
| env | `dev` (δοκιμές) ή `prod` (παραγωγή) |
| ΑΦΜ εκδότη | ΑΦΜ επιχείρησης (demo πάροχος: 619333103) |

Μένουν **μόνο τοπικά** (π.χ. `fiscal_config.config`). Δεν φεύγουν σε τρίτους/browser.

## 3. Βήματα ρύθμισης

1. Επίλεξε λειτουργία «έκδοση μέσω παρόχου».
2. Συμπλήρωσε username/password/activationCode + env.
3. «Δοκιμή σύνδεσης» → επιβεβαιώνει credentials, κατεβάζει λίστες.
4. «Φόρτωση λιστών παρόχου» → γεμίζουν: τύποι παραστατικών, κατηγορίες/χαρακτηρισμοί εσόδου, acquirers, απαλλαγές ΦΠΑ, νομίσματα.
5. Ρύθμισε παραστατικά ([fiscal-documents.md](./fiscal-documents.md)) → «Δοκιμή έκδοσης».

## 4. Κανόνες πληρωμής (PaymentStatus) — ΚΡΙΣΙΜΟ

Όταν δηλώνεται `PaymentStatus`, **πρέπει** να συνοδεύεται από `AcquirerId` + `TidNsp`:

| Τρόπος | PaymentStatus | Acquirer/TidNsp |
|---|---|---|
| Μετρητά | **2** (Με Διασύνδεση) | ναι |
| Κάρτα με πραγματικό POS/Viva transactionId | 2 | ναι (το πραγματικό tid) |
| Κάρτα χωρίς transactionId | 1 | ναι (acquirer + tid) |

Αλλιώς: `1006` «εκτός ορίων» (λείπει/0), `1174` «token μόνο POS» (χωρίς acquirer/tid).

## 5. Αρίθμηση ΑΑ (περίληψη — αναλυτικά στο fiscal-documents.md)

- Επόμενο ΑΑ = `MAX(αα διαβιβασμένων στη σειρά) + 1`. Αποτυχημένες προσπάθειες **δεν** καίνε αριθμούς.
- Προαιρετικό «Αρχικός/Τρέχον ΑΑ» ως πάτωμα· αποθηκεύεται μετά από κάθε έκδοση ώστε να είναι ορατό/ρυθμιζόμενο.
- **Σειρές πάντα σε μία ενιαία γραφή** (π.χ. ελληνικά): «ΑΠΥ» με ελληνικό Υ ≠ «ΑΠY» με λατινικό Y → αλλιώς μηδενίζει/διπλασιάζεται ο αύξων.

## 6. Auto-retry σε διπλό ΑΑ (1114 / 1196) — ΣΗΜΑΝΤΙΚΟ

Αν ο πάροχος έχει **ήδη ΜΑΡΚ** για κάποιο ΑΑ (είναι «μπροστά» από την τοπική αρίθμηση,
π.χ. demo δεδομένα), αυξάνουμε το ΑΑ και ξαναδοκιμάζουμε αυτόματα (έως ~25 φορές).
Καλύπτονται και οι δύο διατυπώσεις:

```
/\b1114\b|\b1196\b|έχει λάβει ήδη|έχει λάβει Μ\.?ΑΡ\.?Κ|ήδη αποθηκευτεί/i
```

Ο **ίδιος βρόχος** εφαρμόζεται και στο πιστωτικό (όχι μόνο στην πώληση).

## 7. Πιστωτικά (ακύρωση)

- Λιανική (Πιστωτικό 11.4 / τύπος 22): **ΧΩΡΙΣ** `CorrelatedMarks`, με **θετικά** ποσά (νέο παραστατικό, όχι αρνητικό) — οδηγία RBS.
- Τοπικά: ακύρωση γραμμών, μείωση τζίρου/ΦΠΑ, αρνητική κίνηση ταμείου.

## 8. Online πωλήσεις

Κατά τον συγχρονισμό (ή «έκδοση εκκρεμών»), εκδίδεται παραστατικό + στέλνεται email με
σύνδεσμο στο επίσημο PDF (ΜΑΡΚ). Πωλήσεις **μηδενικού ποσού** δεν εκδίδουν (δεν καίνε αρίθμηση).

## 9. Πίνακας σφαλμάτων → λύση

| Κωδικός | Αιτία / Λύση |
|---|---|
| 1006 | Λείπει/0 PaymentStatus σε μετρητά → 2 + acquirer + tid |
| 1174 | PaymentStatus χωρίς AcquirerId/TidNsp → πρόσθεσέ τα |
| 1114 | Διπλό ΑΑ (ο πάροχος έχει ΜΑΡΚ) → auto-retry· έλεγξε σειρά/αρχικό ΑΑ |
| 1196 | «…έχει ήδη αποθηκευτεί» (συγκρ. με υπάρχον ΑΑ) → auto-retry· ανέβασε το «Τρέχον ΑΑ» |
| 1192 | Κάρτα τύπος 7 με PaymentStatus 2 χωρίς πραγματικό POS tx → χρησιμοποίησε 1 |
| 1010 (1.5) | 0% ΦΠΑ χωρίς «Κατηγορία απαλλαγής ΦΠΑ» → όρισέ την στο παραστατικό |

Διάγνωση: κράτα το **raw request+response** ανά παραστατικό για έλεγχο.

## 10. Endpoints (αναφορά)

```
POST /api/v1.0/provider/Authorize         { username, password, activationCode } → parentToken
POST /api/v1.0/provider/RefreshToken      (header ParentToken) → bearer token
GET  /api/v1.0/provider/InvoiceTypes | VatCategories | PaymentMethods |
     IncomeCategories | IncomeValues | VatExemptions | Acquirers | Currencies
POST /api/v1.1/provider/PostInvoice1155    (έκδοση → ΜΑΡΚ/QR)
POST /api/v1.1/provider/SendPaymentMethods (2ο βήμα κάρτας/POS)
POST /api/v1.0/provider/invoicevoid        (ακύρωση/void με guid)
```

## Παγίδες που έχουμε ήδη φάει

- **Ομόγραφες σειρές** (λατινικό vs ελληνικό Υ/Α/Ρ…) → μηδενισμός/διπλασιασμός αρίθμησης. Κανονικοποίησε σε μία γραφή + relabel παλιών rows.
- **PaymentStatus χωρίς acquirer/tid** → 1006/1174. Στείλε πάντα και τα τρία.
- **Demo πάροχος «μπροστά»** στην αρίθμηση → 1196. Λύση: auto-retry + ρυθμιζόμενο «Τρέχον ΑΑ».
- **Πιστωτικό με αρνητικά/CorrelatedMarks** → απόρριψη. Στη λιανική: θετικά, χωρίς correlation.
