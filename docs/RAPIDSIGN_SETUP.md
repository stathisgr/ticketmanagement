# Σύνδεση με Πάροχο myDATA — RBS / RapidSign

Οδηγίες σύνδεσης & ρύθμισης του Alpha Ticket Manager με τον πάροχο ηλεκτρονικής
τιμολόγησης **RBS / RapidSign** για αυτόματη έκδοση παραστατικών (ΑΠΥ, ΑΛΠ, Πιστωτικά)
και διαβίβαση στη myDATA (ΑΑΔΕ).

---

## 1. Τι κάνει η σύνδεση

Όταν είναι ενεργή η λειτουργία **«Εισιτήριο μέσω Παρόχου»**, κάθε πώληση (ταμείο & online)
εκδίδει αυτόματα παραστατικό στον πάροχο, παίρνει **ΜΑΡΚ / UID / QR (myDATA)** και τα
αποτυπώνει στο εισιτήριο/απόδειξη. Κάθε ακύρωση εκδίδει **Πιστωτικό**.

- **Hosts:** dev `https://dev.rapidsign.com.gr` · prod `https://app.rapidsign.com.gr`
- **Τεκμηρίωση παρόχου:** `https://api.mymat.com.gr`
- **Ροή auth:** `Authorize` (username/password/activationCode) → parentToken →
  `RefreshToken` (header `ParentToken`) → bearer token για όλες τις κλήσεις.

---

## 2. Διαπιστευτήρια (τα δίνει ο πάροχος)

| Πεδίο | Περιγραφή |
|---|---|
| **username** | Όνομα χρήστη API (RapidSign) |
| **password** | Κωδικός API |
| **activationCode** | Κωδικός ενεργοποίησης εγκατάστασης |
| **env** | `dev` (δοκιμές) ή `prod` (παραγωγή) |
| **ΑΦΜ εκδότη** | Το ΑΦΜ της επιχείρησης (στο demo: 619333103) |

> Τα credentials **δεν** αποθηκεύονται πουθενά αλλού — μόνο τοπικά στη βάση
> (`fiscal_config.config`). Δεν διαβιβάζονται σε τρίτους.

---

## 3. Πού μπαίνουν (βήματα)

1. **Ρυθμίσεις → Εκτυπωτές → Λειτουργία έκδοσης** → επίλεξε **«Εισιτήριο μέσω Παρόχου»**.
2. Συμπλήρωσε **username / password / activationCode** και **env** (dev ή prod).
3. Πάτησε **«Δοκιμή σύνδεσης»** — επιβεβαιώνει credentials και κατεβάζει τις λίστες.
4. Πήγαινε **Ρυθμίσεις → Παραστατικά → «Φόρτωση λιστών παρόχου»** για να γεμίσουν
   οι επιλογές (τύποι παραστατικών, κατηγορίες/χαρακτηρισμοί εσόδου, acquirers, απαλλαγές ΦΠΑ).
5. Ρύθμισε τα παραστατικά (ενότητα 4) και πάτησε **«Δοκιμή έκδοσης ΑΠΥ»**.

---

## 4. Ρύθμιση Παραστατικών (ενιαία λίστα)

Στην καρτέλα **Ρυθμίσεις → Παραστατικά** ορίζεις μια **λίστα παραστατικών**. Κάθε γραμμή έχει:

- **Ρόλος:** Πώληση ή Πιστωτικό
- **Για:** Υπηρεσίες (εισιτήρια) ή Προϊόντα (εμπορικά)
- **Τύπος συναλλαγής:** Λιανική ή Τιμολόγιο *(προς το παρόν εκδίδεται μόνο Λιανική)*
- **Πιστωτικό του:** (στα παραστατικά πώλησης) ποιο πιστωτικό αντιστοιχεί
- **Τύπος myDATA, Σειρά, Αρχικός ΑΑ, Κατηγορία/Χαρακτηρισμός εσόδου, Απαλλαγή ΦΠΑ,
  Πληρωμές (μετρητά/κάρτα), Acquirer, Διασύνδεση POS**

Τυπικά παραστατικά:

| Παραστατικό | Τύπος myDATA | Σειρά | Για | Ρόλος |
|---|---|---|---|---|
| Απόδειξη Παροχής Υπηρεσιών (ΑΠΥ) | 20 (11.2) | ΑΠY | Υπηρεσίες | Πώληση |
| Πιστωτικό Υπηρεσιών (ΠΑΠΥ) | 22 (11.4) | ΠΑΠΥ | Υπηρεσίες | Πιστωτικό |
| Απόδειξη Λιανικής Πώλησης (ΑΛΠ) | 19 (11.1) | ΑΛΠ | Προϊόντα | Πώληση |
| Πιστωτικό Στοιχείο Λιανικής (ΠΛΠ) | 22 (11.4) | ΠΛΠ | Προϊόντα | Πιστωτικό |

Η μηχανή διαλέγει αυτόματα: **υπηρεσία/προϊόν** από τον τύπο είδους, **Λιανική**.
Πρέπει να υπάρχει τουλάχιστον **ένα ενεργό** «Υπηρεσίες · Λιανική» (ΑΠΥ).

---

## 5. Κανόνες πληρωμής (PaymentStatus) — ΚΡΙΣΙΜΟ

Επιβεβαιωμένο από τον πάροχο: όταν δηλώνεται `PaymentStatus`, **πρέπει** να συνοδεύεται
από `AcquirerId` + `TidNsp`. Διαφορετικά:

- `PaymentStatus = 0` ή απουσία → σφάλμα **1006** «εκτός ορίων»
- `PaymentStatus = 1` χωρίς POS data → σφάλμα **1174** «token μόνο POS»

Πώς στέλνει το σύστημα:

| Τρόπος | PaymentStatus | Acquirer/TidNsp |
|---|---|---|
| **Μετρητά** | **2** | ναι (από ρύθμιση/προεπιλογή) |
| **Κάρτα με πραγματικό Viva transactionId** | 2 | ναι (το πραγματικό tid) |
| **Κάρτα χωρίς transactionId** | 1 | ναι (acquirer + tid) |

Στις ρυθμίσεις το πεδίο εμφανίζεται ως **«Διασύνδεση POS»** (2 = Με Διασύνδεση, 1 = Χωρίς).

---

## 6. Αρίθμηση (ΑΑ)

- **Ανά σειρά:** το επόμενο ΑΑ = `MAX(αα όσων έχουν διαβιβαστεί στη σειρά) + 1`.
  Οι **αποτυχημένες** προσπάθειες (χωρίς ΜΑΡΚ) **δεν** καίνε αριθμούς.
- **Τρέχον/Αρχικός ΑΑ (ρυθμιζόμενο):** λειτουργεί ως «πάτωμα» — αν οριστεί μεγαλύτερο, ξεκινά από εκεί.
  **Μετά από κάθε επιτυχή έκδοση** το σύστημα αποθηκεύει το τρέχον ΑΑ πίσω στο config του
  παραστατικού, ώστε να φαίνεται και να ρυθμίζεται από τις Ρυθμίσεις → Παραστατικά.
- **Προσοχή στη γραφή της σειράς:** π.χ. «ΑΠY» με **λατινικό Y** ≠ «ΑΠΥ» με ελληνικό Υ.
  Το σύστημα κανονικοποιεί τις σειρές σε **ελληνικά** (forward + relabel παλιών rows) ώστε να συνεχίζει η αρίθμηση.
- **Auto-retry 1114 / 1196:** αν ο πάροχος έχει ήδη ΜΑΡΚ για κάποιο ΑΑ ή «έχει ήδη αποθηκευτεί»
  (είναι «μπροστά», π.χ. demo δεδομένα), το σύστημα αυξάνει το ΑΑ και ξαναδοκιμάζει αυτόματα
  (σε πωλήσεις **και** πιστωτικά).

---

## 7. Πιστωτικά (ακύρωση)

- **Λιανική (Πιστωτικό 11.4 / τύπος 22):** ΧΩΡΙΣ `CorrelatedMarks`, με **θετικά** ποσά
  (οδηγία RBS — νέο παραστατικό, όχι αρνητικό).
- Στην ακύρωση γίνεται και πλήρης τοπικός αντιλογισμός (ακύρωση εισιτηρίων, μείωση
  τζίρου/ΦΠΑ, αρνητική κίνηση ταμείου).
- Τα προϊόντα ακυρώνονται με το **ΠΛΠ**, οι υπηρεσίες με το **ΠΑΠΥ**.

---

## 8. Online πωλήσεις

Κατά τον **συγχρονισμό** (ή με «Έκδοση εκκρεμών online ΑΠΥ»), εκδίδεται ΑΠΥ στον πάροχο
και στέλνεται 2ο email στον πελάτη με σύνδεσμο στο επίσημο PDF (ΜΑΡΚ). Πωλήσεις
**μηδενικού ποσού** δεν εκδίδουν παραστατικό (δεν «καίνε» αρίθμηση).

---

## 9. Σφάλματα παρόχου — Αντιμετώπιση

| Κωδικός | Μήνυμα | Αιτία / Λύση |
|---|---|---|
| **1006** | «κατάσταση πληρωμής εκτός ορίων» | Λείπει/0 το PaymentStatus σε μετρητά → στείλε 2 + acquirer + tid |
| **1174** | «token πληρωμής μόνο μέσω POS» | PaymentStatus χωρίς AcquirerId/TidNsp → πρόσθεσέ τα (μετρητά: 2 + acquirer + tid) |
| **1114** | «το παραστατικό X-N έχει ήδη ΜΑΡΚ» | Διπλό ΑΑ → καλύπτεται από το auto-retry· έλεγξε σειρά/αρχικό ΑΑ |
| **1196** | «το παραστατικό X-N έχει ήδη αποθηκευτεί» | Το τοπικό ΑΑ συγκρούεται με υπάρχον στον πάροχο → auto-retry· ανέβασε το «Τρέχον ΑΑ» στη σειρά |
| **1192** | (κάρτα τύπος 7) | Κάρτα με PaymentStatus 2 χωρίς πραγματικό POS transaction → χρησιμοποίησε 1 |
| **1010 (1.5)** | πεδίο VatExcCatId εκτός ορίων | Σε 0% ΦΠΑ απαιτείται «Κατηγορία απαλλαγής ΦΠΑ» στο παραστατικό |

Διάγνωση: στη σελίδα **Παραστατικά** κάθε γραμμή έχει «απάντηση» (raw payload+response)
για έλεγχο.

---

## 10. Endpoints παρόχου (αναφορά)

```
POST /api/v1.0/provider/Authorize            { username, password, activationCode } → parentToken
POST /api/v1.0/provider/RefreshToken         (header ParentToken) → bearer token
GET  /api/v1.0/provider/InvoiceTypes | VatCategories | PaymentMethods |
     IncomeCategories | IncomeValues | VatExemptions | Acquirers | Currencies
POST /api/v1.1/provider/PostInvoice1155       (έκδοση παραστατικού → ΜΑΡΚ/QR)
POST /api/v1.1/provider/SendPaymentMethods    (2ο βήμα κάρτας/POS)
POST /api/v1.0/provider/invoicevoid           (ακύρωση/void με guid)
```

---

*Σημείωση ασφαλείας: τα credentials/κλειδιά καταχωρούνται από τον χρήστη και μένουν
τοπικά. Για παραγωγή, άλλαξε `env` σε `prod` και χρησιμοποίησε τα πραγματικά στοιχεία RBS.*
